# Дашборд по воронкам и тестам — спецификация для аналитика

**Дата:** 2026-07-23
**Статус:** черновик для обсуждения с аналитиком, не финальное ТЗ
**Автор:** Артём Пак (Growth PM, Cleverside)

---

## 1. Зачем это нужно

Сейчас данные по воронкам, тестам и спенду разбросаны по ClickHouse (события), Chargebee (платежи, activation, churn) и Google Sheets (спенд по кампаниям — вручную заполняемая вкладка). Собирать картину по каждому тесту приходится руками: JOIN'ить `gbUserId` из web_events с Chargebee, тянуть спенд из отдельной вкладки, считать CPA/ROAS в голове.

Цель дашборда — автоматизировать эту сборку, чтобы:
- ежедневно видеть, как ведут себя воронки (доходимость по шагам, CPA/ROAS);
- по ходу A/B-теста принимать решение scale/kill/iterate, не дожидаясь ручного анализа;
- раз в неделю показывать CEO верхнеуровневую картину;
- разбирать конкретные гипотезы ad-hoc, когда что-то непонятно в цифрах.

Дашборд должен работать не только на примере одной воронки, а масштабироваться на новые воронки и тесты по мере их запуска — это встроено в дизайн через фильтр «Воронка / эксперимент», а не хардкод.

**Аудитория:** я (Growth PM) + аналитик, который будет строить и поддерживать дашборд + команда, которая будет на него смотреть. Формулировки и подписи должны быть понятны без моих устных пояснений.

## 2. Общая структура

Вертикальная портянка (по образцу привычных BI-дашбордов), сверху вниз:

1. **Глобальные фильтры** — закреплены сверху, влияют на все разделы ниже.
2. **Общий обзор (Overview)** — раскрываемая таблица Воронка → Эксперимент → Группа → Тарифный план по экономике/выручке/конверсиям для текущего выбора фильтров. Идёт первым — общая картина перед деталями.
3. **Конверсия в покупку** — WelcomeDone → Paywall → Subscription, по каждому эксперименту/группе. График в одной панели + таблица в другой.
4. **Конверсия в покупку от старта эксперимента** — та же конверсия, но по когортному дню (день 1, 2, 3...), с ARPU и разбивкой по тарифным планам справа вместо WelcomeDone/Paywall.
5. **Retention / Churn по дням** — что происходит с подписавшимися на 3-й/7-й/14-й день. График в одной панели + таблица в другой.
6. **Retention / LTV (расчётный)** — LTV на 30/60/90 дней, payback period. (раздел из первой итерации, будет дорабатываться отдельно)
7. **Сегменты** — отдельная вкладка (не часть общего скролла): кто приходит / покупает / остаётся, по полу, возрасту, психографике (ответы в квизе).

Разделы 2–6 — общий скролл на одном экране. Раздел 7 — отдельная вкладка, потому что это другой тип вопроса («кто эти люди», а не «как работает воронка»).

**Статус по итерациям:** «Общий обзор» переделан из hero-цифр в раскрываемую таблицу, добавлен раздел «Конверсия от старта эксперимента», разделы «Конверсия в покупку»/«Retention Churn» получили визуально раздельные панели график/таблица, старый отдельный раздел «CPA/ROAS» убран как дублирующий (полностью покрыт «Общим обзором»). «Retention/LTV расчётный» из первой итерации пока остаётся черновиком.

## 3. Глобальные фильтры

Расположены в шапке, применяются ко всем разделам ниже. Два уровня иерархии — **воронка** и **эксперимент** — это разные сущности, их нельзя схлопывать в один фильтр:

- **Воронка** — конкретный продукт/лендинг, например `funnel-a.example.com` или `funnel-b.example.com`. Дальше будут добавляться новые воронки.
- **Эксперимент** — конкретный A/B-тест внутри выбранной воронки. Список экспериментов в дропдауне **зависит от того, какая воронка выбрана** (если воронка не выбрана — показываются эксперименты всех воронок). Название эксперимента — то, что реально стоит в кампании/тесте (например, название campaign из вкладки `spend`), а не придуманное.
  - Один эксперимент может содержать **несколько групп** (control / variant), если тест был A/B. Тогда на графиках ниже это несколько линий одного цвета-семейства, в таблице — несколько строк с одинаковым названием эксперимента и разными названиями групп.
  - Если эксперимент не выбран — все графики и таблицы ниже показывают **все эксперименты** выбранной воронки (или вообще все, если воронка тоже не выбрана) построчно/полиниям, как в общем обзоре.
- **Версия** — отдельный фильтр рядом с «Эксперимент», на случай если один и тот же эксперимент запускался несколько раз итерациями (v1, v2, v3...) — например, тест доработали и перезапустили через несколько дней. Это не то же самое, что «группа» (control/variant внутри одного запуска): версия — это про **разные запуски во времени**, группа — про **разные варианты внутри одного запуска**. Список версий в дропдауне зависит от выбранного эксперимента (и воронки). Если у эксперимента только одна версия — фильтр неактивен (нет смысла его использовать).
- **Шкала** — выбирается первой, до периода: **календарная** (тренд по датам) или **день теста** (день 1, день 2... от старта конкретного эксперимента — нужно, чтобы сравнивать тесты, стартовавшие в разные даты, на одной шкале).
- **Период** — диапазон по дням внутри выбранной шкалы (календарные даты или дни 1–N от старта). Быстрые пресеты 7д / 30д / весь тест.
- **Источник трафика / кампания** — Facebook campaign и т.п. Сейчас весь трафик — Facebook, фильтр пока с одним значением, закладываем на будущее, когда каналов станет больше.

Сегментные фильтры (пол, возраст, психография) — **не в общей шапке**, а внутри вкладки «Сегменты», см. раздел 9.

**Открытый вопрос из практики:** нередко бывает так, что две «группы» на деле — это одна и та же кампания за один и тот же период, то есть технически это **одна воронка/эксперимент с двумя группами**, а не два отдельных эксперимента. В прототипе показан именно такой случай (Experiment Alpha, Group 1 / Group 2) — при заполнении реальными данными это нужно перепроверять по вкладке спенда/кампаний, а не полагаться на название теста.

## 4. Раздел «Общий обзор» (Overview)

Первое, что видно под фильтрами — не hero-цифры, а **раскрываемая таблица** с иерархией **Воронка → Эксперимент → Группа → Тарифный план**. У строк верхнего уровня — стрелка ▸/▾: разворачивает следующий уровень вложенности. Если наверху в фильтрах уже выбрана конкретная воронка или эксперимент — таблица сразу стартует с этого уровня (не показывает уже зафиксированный фильтром уровень отдельной строкой).

**Колонки** (первая — название в рамке, дальше числа):

| Название | Источник | Spend | CPA 0-day | CPA (опл.) | ROAS 0d | Gross Rev. | Net Rev. | ARPU D3 | ARPU D14 | Start→Триал | Триал→Опл. | Churn D3 | Churn D14 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|

- **Название** — воронка / эксперимент / группа (control-variant) / тарифный план, в зависимости от уровня вложенности. Показывается в визуально выделенной рамке.
- **Источник** — рекламный канал/кампания. На верхних уровнях (воронка, эксперимент), если под ними несколько разных источников — показывается «N источника» вместо конкретного названия.
- **Spend / CPA 0-day / CPA (опл.) / ROAS 0d** — экономика, агрегированная по всем строкам ниже текущего уровня.
- **Gross Rev. / Net Rev. / ARPU D3 / ARPU D14** — выручка, см. раздел 12 «Словарь метрик» за формулами.
- **Start→Триал / Триал→Опл.** — конверсии по воронке.
- **Churn D3 / Churn D14** — retention. День 30 в overview пока не включаем — оставили только 3 и 14, чтобы не перегружать таблицу (день 30 доступен в разделе «Retention / Churn по дням» при необходимости).
- На уровне **тарифного плана** часть колонок не применима (Spend/CPA/ROAS/конверсии/churn считаются на уровне группы или выше, не на уровне плана) — в этих ячейках стоит прочерк, а не 0.

**Форма:** одна широкая таблица с горизонтальным скроллом на узких экранах, а не карточки. Точные формулы всех метрик — в разделе 12 «Словарь метрик».

**Целевые ориентиры** (из текущего growth-плана команды, показывать как референсную линию/порог там, где уместно, например подсветкой строки/ячейки):
- ROAS 0d ≥ целевого значения (пример: 40%)
- Конверсия Session Start → покупка ≥ целевого значения (пример: 2.5%)

**Про сравнение групп/экспериментов между собой:** сравнение делается через раскрытие дерева и фильтр «эксперимент» (раздел 3), а не отдельным «baseline vs variant» экраном — если у эксперимента несколько групп, они просто идут рядом строками при разворачивании. Остаётся открытым вопрос — нужно ли добавить индикатор статистической значимости различий между группами (важно, пока часть тестов идёт на выборке в единицы покупателей) — уточнить у аналитика, возможно ли это технически корректно посчитать при таком объёме.

## 5. Раздел «Конверсия в покупку» (WelcomeDone → Paywall → Subscription)

Визуально раздел состоит из **двух отдельных панелей** внутри одной секции — график и таблица не просто рядом в общей сетке, а каждый в своей рамке/карточке («два окна»).

**Левая панель — линейный график.** По оси X — период (в выбранной шкале: календарная дата или день теста), по оси Y — % конверсии в покупку (Subscription / Session Start). Одна линия на каждую группу текущего выбора:
- если выбран конкретный эксперимент — по линии на каждую его группу (control/variant, если их несколько);
- если эксперимент не выбран (только воронка или вообще ничего) — по линии на каждый эксперимент/группу в разрезе.

**Правая панель — таблица** (построчно те же сущности, что и линии слева), в формате плоского списка с сортировкой по CVR (по образцу референса, который скинул Артём — простая таблица, без бар-чартов внутри ячеек):

| Эксперимент / группа | WelcomeDone, n | WelcomeDone, % | Paywall, n | Paywall, % | WD → Paywall, % | Subscription, n | Subscription, % |
|---|---|---|---|---|---|---|---|

- **n** — абсолютное число юзеров, дошедших до чекпоинта.
- **%** — конверсия от Session Start (или от предыдущего шага — уточнить у аналитика, какой вид привычнее для чтения; в текущей практике команды используется % от Session Start).
- **WD → Paywall, %** — отдельная колонка для конверсии именно с 2-го экрана на пейвол (Paywall% / WelcomeDone%) — этот шаг регулярно проверяется отдельно от общей воронки, поэтому вынесен явной колонкой, а не только виден расчётом в голове.
- Если наверху выбран конкретный эксперимент — в таблице остаются только его группы. Если нет — таблица показывает все эксперименты/группы выбранной воронки (или вообще все).

Чекпоинты соответствуют событийной модели: WelcomeDone = 2-й экран (первое действие после старта квиза), Paywall = дошёл до экрана оплаты (CompleteRegistration), Subscription = купил (Subscribe).

## 6. Раздел «Конверсия в покупку от старта эксперимента»

Та же логика «график слева / таблица справа, каждый в своей панели», но с другим фокусом. В этом разделе **две панели с графиками** (не одна) плюс таблица:

**Панель 1 — конверсия.** По оси X — **день с момента старта эксперимента** (когорта, день 1, день 2...), а не календарная дата — так тесты, стартовавшие в разные даты, сравнимы на одной шкале. По оси Y — % конверсии в покупку, так же как в разделе 5.

**Панель 2 — ARPU, накопительно.** По оси X — контрольные точки D1 / D3 / D7 / D14 от старта. По оси Y — накопленная выручка на одного купившего триал (buyer) — та же величина, что в колонках ARPU D3/D14 общего обзора (раздел 4), но как кривая, а не одна цифра, чтобы видеть форму накопления, а не только конечную точку.

**Таблица** (под графиками, во всю ширину), вместо WelcomeDone/Paywall здесь **выручка и тарифные планы**:

| Эксперимент / группа | Активаций, n | ARPU (D14) | Monthly, n | Monthly, revenue | Annual, n | Annual, revenue |
|---|---|---|---|---|---|---|

- Смысл раздела — не просто «дошли ли до покупки», а «сколько реально заработали, как быстро и на каком тарифе» для той же когорты.
- ARPU (D14) в таблице считается на буквально ту же базу (все купившие триал, buyers), что и конечная точка графика ARPU — числа должны совпадать, это не два независимых расчёта.
- Разбивка по тарифным планам — открытый вопрос: пока заложены Monthly / Annual как пример, точный список тарифов даст аналитик по факту того, что реально продаётся на пейволе.

## 7. Раздел «Retention / Churn по дням»

Тот же принцип «две панели» (график слева, таблица справа), что и в разделах 5–6.

**Левая панель — линейный график.** По оси X — контрольные точки D3 / D7 / D14 от момента подписки, по оси Y — % отменивших подписку (churn, накопительно). Одна линия на каждую группу — та же логика фильтрации, что и в разделах 5–6 (эксперимент выбран → его группы; не выбран → все эксперименты воронки/все воронки).

**Правая панель — таблица**, тот же принцип (плоский список, сортировка):

| Эксперимент / группа | Subscriptions, n | Churn D3, % | Churn D7, % | Churn D14, % |
|---|---|---|---|---|

**Важная оговорка:** свежие эксперименты (например, тесты последней недели) не успевают дойти до D14 — в этих ячейках должен быть явный статус «недостаточно данных», а не 0%, потому что это right-censoring, а не факт хорошего retention. Та же логика применяется к Churn D14 в общем обзоре (раздел 4).

## 8. Раздел «Retention / LTV» (черновик первой итерации — будет пересмотрен)

Расчётный, не просто % активации/отмены — нужен для принятия решений по payback и unit-экономике.

**Метрики:**
- % активации (первая оплата) и % итоговых отмен — как базовый слой
- **LTV на горизонте 30 / 60 / 90 дней** — прогноз накопленной выручки на подписчика с фиксированной даты старта подписки
- **Payback period к целевому ROAS 0d** (пример порога: 40%) — сколько дней нужно, чтобы CPA окупился при текущей динамике renewals
- **LTV:CAC ratio** — здоровое соотношение (ориентир, например 3:1)

**Важная оговорка для аналитика:** свежие когорты (тесты последней недели) будут показывать 0 renewals не потому что плохой retention, а потому что просто не прошёл полный биллинг-цикл (right-censoring). Дашборд должен явно маркировать такие когорты как «данных недостаточно», а не показывать 0% как факт.

**Форма:** line chart накопленного LTV по дням (с горизонтом 30/60/90 как вертикальные референсные линии) + hero-цифры payback period и LTV:CAC.

## 9. Вкладка «Сегменты»

Отдельная вкладка, не часть общего скролла. Отвечает на вопрос «кто эти люди», а не «как работает воронка».

**Свои фильтры** (не глобальные): пол, возраст, психография / ответы на конкретные вопросы квиза.

**Разделы:**
- Кто приходит (распределение по полу/возрасту/психографии на входе в воронку)
- Кто покупает (то же самое, но срез по купившим)
- Кто остаётся (то же самое, срез по активировавшимся/не отменившим)

**Форма:** горизонтальные bar chart по срезам, с возможностью переключения между «приходит / покупает / остаётся» одним тумблером (не три отдельных постоянно видимых блока — чтобы не перегружать экран).

## 10. Данные и источники (для аналитика)

| Метрика | Источник | Комментарий |
|---|---|---|
| Session Start | внешний (лендинг) | нет в нашей БД, берётся из Google Sheets/внешнего трекера |
| ViewContent, LevelCompleted_N, CompleteRegistration, Subscribe | ClickHouse `web.web_events`, JSON `content.*` | склейка сессий по `gbUserId` |
| Spend, trials (по кампании) | Google Sheets, вкладка `spend` (ручное заполнение) | **первый кандидат на автоматизацию** — сейчас данные приходится вручную выгружать из FB Ads в таблицу |
| subscription_created / activated / renewed / cancelled | ClickHouse `web.chargebee_events` | склейка через `customer.meta_data.user_id` = `gbUserId` |

**Самый большой разрыв в автоматизации сейчас** — спенд по кампаниям существует только в Google Sheets и заполняется вручную. Если дашборд должен обновляться без моего участия, спенд нужно тянуть напрямую из Facebook Ads API (или другого источника кампаний), а не из таблицы. Это отдельный вопрос к аналитику: возможно ли подключить рекламные API как источник, или пока остаёмся на ручной вкладке с ручным импортом в BI.

## 11. Словарь метрик

Что значит каждая метрика на дашборде и как она считается — чтобы у меня, аналитика и команды было одно понимание, без устных уточнений.

| Метрика | Что значит | Как считаем |
|---|---|---|
| Session Start | Заход на лендинг / старт сессии. | Внешнее событие, в нашей БД нет — берётся из внешнего трекера лендинга/Google Sheets. |
| WelcomeDone (2-й экран) | Юзер начал квиз — первое действие после лендинга (клик «начать»/выбор пола). | `COUNT(DISTINCT session)` по событию `ViewContent` в `web.web_events`. |
| Paywall (CompleteRegistration) | Дошёл до экрана оплаты. | `COUNT(DISTINCT session)` по событию `CompleteRegistration`. |
| Триал / Subscription | Оформил пробную подписку на пейволе — ещё не значит оплатил. | `COUNT(DISTINCT session)` по событию `Subscribe` в `web_events` = `COUNT` по `subscription_created` в `chargebee_events` (склейка по `gbUserId`). |
| Первая оплата / Активация | Реально прошло первое списание — юзер не отменил в течение триала. | `COUNT` по событию `subscription_activated` в `chargebee_events`. |
| CPA 0-day (по триалу) | Стоимость привлечения одного оформившего триал — ещё без гарантии, что оплатит. | Spend / кол-во триалов (`Subscribe`). |
| CPA (реальный, по 1-й оплате) | Реальная стоимость платящего юзера — с поправкой на то, что часть триалов отменяется до оплаты. Основная метрика для scale/kill. | Spend / кол-во активаций (`subscription_activated`). |
| ROAS 0-day | Мгновенный возврат на рекламные расходы в момент оформления триала (не итоговый ROAS). | (кол-во триалов × цена триала) / Spend. |
| Gross Revenue | Вся выручка, начисленная по подпискам за период, без вычетов. | Сумма успешных списаний: `subscription_activated` + `subscription_renewed` × сумма платежа. |
| Net Revenue | Выручка за вычетом комиссий платёжной системы, налогов и chargebacks/disputes. | Gross Revenue − комиссии/налоги − сумма спорных и возвращённых платежей. |
| ARPU D3 / D14 | Средняя выручка на одного купившего триал (buyer) к 3-му/14-му дню с момента старта триала. | Накопленная выручка когорты на день N / кол-во купивших триал (buyers) в когорте. |
| Конверсия Session Start → Триал | Доля дошедших от захода на лендинг до оформления триала. | Триалы / Session Start. |
| Конверсия Триал → Первая оплата | Доля триалов, которые не отменились и дошли до первого списания. | Активации / Триалы. |
| Churn D3 / D7 / D14 | Накопленная доля отписавшихся к N-му дню от момента активации подписки. | `COUNT(subscription_cancelled` к дню N`)` / `COUNT(subscription_activated)`. |
| LTV D30 / D60 / D90 | Прогнозная накопленная выручка на одного активировавшегося юзера к дню N. | Сумма выручки когорты на день N / кол-во активировавшихся в когорте. |
| Payback period | Сколько дней нужно, чтобы накопленная выручка на юзера окупила CPA. | Минимальный день N, при котором LTV(N) ≥ CPA (по 1-й оплате). |
| LTV:CAC | Соотношение прогнозной выручки к стоимости привлечения — индикатор здоровой unit-экономики. | LTV (на выбранном горизонте, напр. D90) / CPA. |

Таблица — рабочий черновик (примера с прошлого проекта под рукой не нашлось), но структура «что значит / как считаем» пусть останется зафиксированной как формат. Список пополняется по мере появления новых метрик на дашборде.

## 12. Открытые вопросы к аналитику

- Реалистичный источник и способ автоматической подгрузки спенда (FB Ads API / другое) — без этого дашборд не будет полностью самостоятельным.
- Какой метод статзначимости использовать для конверсий с маленькой выборкой (сейчас часть тестов — единицы покупателей) — возможно, часть сравнений технически нельзя корректно проверить на значимость, и это нужно явно показывать пользователю.
- Периодичность обновления данных (real-time / раз в день) — зависит от возможностей источника спенда.
- Нужно ли ограничивать доступ к дашборду (сегменты по полу/возрасту — не чувствительные данные, но всё же стоит формализовать, кто видит какие разделы).
